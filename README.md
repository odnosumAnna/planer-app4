# Planer - Стратегія роботи з даними

## Локальне сховище
Використовується **Room** (SQLite) для зберігання всіх даних:
- Tasks (задачі)
- Categories (категорії)
- Users (користувачі)

## API Контракт (REST)

| Метод | Endpoint | Опис | Body/Params |
|-------|----------|------|-------------|
| GET | /api/tasks | Отримати всі задачі | - |
| GET | /api/tasks/{id} | Отримати задачу за ID | - |
| POST | /api/tasks | Створити нову задачу | `{title, description, deadline, category}` |
| PUT | /api/tasks/{id} | Оновити задачу | `{title, description, isCompleted, deadline, category}` |
| DELETE | /api/tasks/{id} | Видалити задачу | - |
| GET | /api/categories | Отримати всі категорії | - |
| GET | /api/user/profile | Отримати профіль | - |

## Стратегія комунікації: **Offline-first**

### Принцип роботи:
1. **Читання** - завжди з локальної бази даних (Room)
2. **Запис** - спочатку в локальну БД зі статусом `PENDING`, потім фонова синхронізація з сервером
3. **Синхронізація** - при відновленні мережі або pull-to-refresh
4. **Статуси синхронізації**:
   - `SYNCED` - дані синхронізовані з сервером
   - `PENDING` - очікують відправки
   - `ERROR` - помилка при синхронізації

### Чому Offline-first:
- Користувач завжди має доступ до своїх даних, навіть без інтернету
- Миттєвий відгук UI (немає очікування сервера)
- Дані не губляться при поганому з'єднанні
- Підходить для персонального планувальника

### Repository Pattern:
- Інкапсулює логіку роботи з даними
- Об'єднує локальне сховище (Room) та API-шар (Retrofit/Mock)
- Забезпечує єдине джерело правди для UI

### Що відбувається без інтернету:
1. Всі операції CRUD працюють локально
2. Дані зберігаються зі статусом `PENDING`
3. При відновленні мережі автоматично синхронізуються
4. Користувач бачить статус синхронізації в UI

## 🔐 Біометрична автентифікація

Додаток підтримує біометричну автентифікацію для безпечного доступу.

### Можливості:
- Підтримка Fingerprint, Face ID, Iris
- Перевірка наявності датчика
- Збереження налаштувань користувача
- Екран блокування з автоматичним запитом
- Підтвердження небезпечних операцій

### Стани автентифікації:
- Idle - очікування
- Authenticating - процес сканування
- Success - успішно
- Failed - помилка
- Unavailable - датчик недоступний
